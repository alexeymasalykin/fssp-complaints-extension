# Техническое задание: Chrome-расширение для автоматизации проверки по РКЛ через Госуслуги

**Версия:** 1.2 (обновлено: Фаза 2 завершена, добавлена спецификация админ-панели)
**Дата:** 04.03.2026
**Рабочее название продукта:** RKL Check

---

## 1. Описание продукта

Браузерное расширение для Google Chrome (и Chromium-браузеров: Яндекс.Браузер, Edge), автоматизирующее массовую проверку иностранных сотрудников по Реестру контролируемых лиц (РКЛ) МВД через портал Госуслуг.

Расширение работает **локально в браузере** пользователя. Персональные данные сотрудников **не покидают контур организации** --- передаются только на Госуслуги через авторизованную сессию, без промежуточных серверов.

**Бизнес-модель:** подписка (SaaS). Расширение распространяется через Chrome Web Store (unlisted --- по прямой ссылке). Функционал активируется лицензионным ключом, привязанным к подписке.

## 4. Архитектура

### 4.1. Обзор компонентов

Система состоит из двух независимых частей:

**Часть 1 --- Chrome-расширение (клиентская):**

- Popup UI --- интерфейс управления (загрузка данных, запуск, результаты)
- Content Script --- робот, работающий на странице Госуслуг (определение шагов, чтение результатов; заполнение полей и клики --- через MAIN world из background)
- Background Service Worker --- координатор (очередь, состояние, связь между popup и content script)

**Часть 2 --- Сервер лицензий (серверная):**

- Проверка и активация лицензионных ключей
- Счётчик использования
- Управление тарифами и подписками
- Интеграция с платёжной системой

### 4.3. Принцип работы расширения

Расширение **не перехватывает API-запросы** Госуслуг и **не эмулирует HTTP-запросы**. Оно работает исключительно через DOM: заполняет поля формы, нажимает кнопки, считывает текст результата. Это полный эквивалент ручной работы кадровика, но автоматизированный.

Госуслуги используют **Angular + EPGU-компоненты** (НЕ React). Страница проверки РКЛ --- это SPA (Single Page Application). URL не меняется между шагами формы: `https://www.gosuslugi.ru/655781/1/form`. Переходы между шагами происходят через перерисовку DOM без перезагрузки страницы.

**Двухуровневая модель выполнения:**

- **Content script (ISOLATED world)** --- только чтение: определение текущего шага формы, чтение результата проверки
- **Background → chrome.scripting.executeScript({ world: 'MAIN' })** --- все операции записи: заполнение полей, клики по кнопкам

Content script работает в изолированном мире Chrome. События, созданные из ISOLATED world (`dispatchEvent`), являются untrusted (`isTrusted=false`) и **игнорируются Angular**. Поэтому все DOM-манипуляции, которые должны быть восприняты Angular, выполняются через `executeScript` в MAIN world --- контексте самой страницы.

### 4.4. Разделение данных

**На сервере лицензий хранятся ТОЛЬКО:**

- ID клиента (организации)
- Лицензионный ключ (сгенерированный UUID)
- Название тарифа и дата окончания подписки
- Счётчик выполненных проверок за текущий период
- Email для связи

**Никаких персональных данных сотрудников. Никогда. Ни в каком виде.**

**Локально в браузере (chrome.storage.local / IndexedDB):**

- Загруженные данные сотрудников из Excel/XML (до закрытия сессии или ручной очистки пользователем)
- Результаты проверок текущей сессии
- История проверок (опционально, включается пользователем)
- Настройки расширения (задержка между проверками, формат экспорта)
- Кэш лицензии (для работы при кратковременной потере связи с сервером)

## 5. Подробное описание Госуслуг --- страница проверки РКЛ

### 5.1. URL и навигация

- Адрес сервиса: `https://www.gosuslugi.ru/655781/1/form`
- Тип: SPA на **Angular + EPGU-компоненты** (НЕ React)
- EPGU-компоненты: `lib-button`, `epgu-constructor-screen-buttons`, `lib-date-picker`
- Angular-атрибуты в DOM: `_ngcontent-ng-c*`
- URL не меняется при переходе между шагами формы
- Авторизация: пользователь должен быть залогинен на Госуслугах как сотрудник организации (без ЭП, обычная учётка)
- В шапке отображается "Организациям" --- подтверждение что сессия от имени юрлица

### 5.2. Шаги формы

**ВАЖНО:** фактический порядок шагов --- **intro → birthdate → document → result** (НЕ document → birthdate, как предполагалось изначально).

Между шагами бывают промежуточные loading-страницы ("Загружается", "Подождите", "Проверка данных") --- обрабатываются как `unknown`, ожидание и повтор.

**Шаг 0 --- Intro**

Текст: "Поиск иностранца" / "реестр"

Кнопка: "Начать"

**Шаг 1 --- Дата рождения**

Заголовок: "Укажите дату рождения проверяемого иностранца"

Поля:
- "Дата" (НЕ "Дата рождения") --- поле даты с иконкой календаря, `name="c_birth_date"` (обязательное)
- Чекбокс: "В документе неполная дата рождения"

Розовый блок-предупреждение: "Проверьте данные иностранца. Они должны точно совпадать с документом, удостоверяющим его личность"

Кнопка: "Продолжить"

**Шаг 2 --- Данные документа**

Заголовок страницы: "Укажите данные документа, удостоверяющего личность проверяемого иностранца"

Поля:
- "Серия" --- текстовое поле, подпись "При наличии" (необязательное)
- "Номер" --- текстовое поле, подпись "Как в графе Passport No документа" (обязательное)
- "Дата выдачи" --- поле даты с иконкой календаря, подпись "Как в графе Issue date документа" (обязательное)

Розовый блок-предупреждение: аналогичный шагу 1

Кнопка: "Продолжить"

**Шаг 3 --- Результат**

Варианты ответа:
- "Отсутствует в реестре контролируемых лиц" (с зелёной иконкой паспорта)
- "Состоит в реестре контролируемых лиц" (подтверждено при тестировании)

Информация в результате:
- Документ: [серия+номер], выдан [дата]
- Дата рождения: [дата]
- Данные получены из системы МВД России [дата] [время] МСК

Кнопки:
- "Проверить ещё" --- возврат к intro/birthdate (без перезагрузки страницы)
- "На главную"

### 5.3. Технические наблюдения (обновлено по результатам разработки)

- Фронтенд на **Angular** с EPGU-компонентами (`lib-date-picker`, `lib-button`)
- Поля ввода --- Angular `DefaultValueAccessor` слушает `(input)` event
- `el.value = val` + `dispatchEvent` из ISOLATED world **НЕ работает** --- Angular игнорирует untrusted события
- **Рабочий метод:** `document.execCommand('insertText')` в MAIN world --- создаёт trusted input events
- Поле даты рождения: `name="c_birth_date"`, label "Дата" (не "Дата рождения")
- Время ответа МВД: от 5 до 30 секунд (на тесте 03.03.2026: ответ за ~10 сек)
- Капча отсутствует (авторизованная сессия)
- **Rate limit:** при слишком частых запросах (задержка < 10 сек) Госуслуги показывают страницу "Превышено количество запросов". Рекомендуемая задержка ≥ 10 секунд
- При незаполненных обязательных полях Angular-форма не позволяет перейти на следующий шаг (сообщение "Обязательно для заполнения" у поля)

## 6. Пользовательский сценарий (User Flow)

### 6.1. Первый запуск

1. Пользователь оплачивает подписку на лендинге
2. Получает лицензионный ключ (email + страница подтверждения)
3. Переходит по ссылке в Chrome Web Store, устанавливает расширение
4. Открывает popup расширения (клик по иконке в toolbar)
5. Вводит лицензионный ключ, нажимает "Активировать"
6. Расширение отправляет ключ на сервер
7. Сервер подтверждает: подписка активна, тариф "Бизнес", лимит 6000, использовано 0
8. Интерфейс расширения разблокирован

### 6.2. Подготовка к проверке

1. Пользователь авторизуется на gosuslugi.ru как сотрудник организации
2. Переходит на `gosuslugi.ru/655781/1/form` (или расширение может открыть эту страницу автоматически по кнопке "Открыть Госуслуги")
3. Открывает popup расширения
4. Загружает файл с данными сотрудников (drag-drop область или кнопка "Выбрать файл")
5. Расширение парсит файл (SheetJS для Excel, DOMParser для XML)
6. Отображает таблицу предпросмотра: N строк, колонки серия/номер/дата выдачи/дата рождения/ФИО
7. Подсвечивает ошибки: пустые обязательные поля, некорректный формат дат
8. Показывает информацию: "Найдено N сотрудников. Лимит: использовано X из Y"

### 6.3. Формат входных данных

**Excel (.xlsx) --- основной формат:**

Обязательные колонки (определяются по заголовку первой строки):

| Колонка | Обязательность | Формат | Примечание |
|---|---|---|---|
| Серия | Нет | Текст/число | Может быть пустой |
| Номер | Да | Текст/число | Как в графе Passport No |
| Дата выдачи | Да | ДД.ММ.ГГГГ | Как в графе Issue date |
| Дата рождения | Да | ДД.ММ.ГГГГ | |
| ФИО | Нет | Текст | Не передаётся в Госуслуги, только для отчёта |

Расширение должно уметь распознавать колонки по нескольким вариантам названий:
- "Номер" / "Номер документа" / "Passport No" / "docnum"
- "Дата выдачи" / "Issue date" / "docdate"
- "Дата рождения" / "birthdate" / "ДР"
- "Серия" / "Series"
- "ФИО" / "Имя" / "Name" / "fio"

### 6.4. Процесс проверки --- подробный алгоритм

**Предпроверки:**

1. Активная вкладка содержит URL `gosuslugi.ru/655781`? Если нет --- показать предупреждение "Откройте страницу проверки РКЛ на Госуслугах" с кнопкой "Открыть"
2. Лицензия активна? Если нет --- показать "Подписка истекла" с кнопкой "Продлить"
3. Лимит не исчерпан? Если исчерпан --- показать "Лимит проверок исчерпан (X из X). Повысьте тариф"
4. Файл загружен и валиден? Если нет --- показать ошибки валидации

**Цикл проверки (адаптивный --- для каждого сотрудника):**

Алгоритм использует адаптивный `while` loop, который обрабатывает шаги **в любом порядке**. Между шагами бывают loading-страницы, которые обрабатываются повторным опросом.

```
WHILE (не все шаги пройдены AND state === 'running'):

  1. ОПРЕДЕЛЕНИЕ ШАГА (через content script, ISOLATED world)
     - Прочитать document.body.innerText
     - Regex-матчинг:
       /укажите\s+данные\s+документа/i -> 'document'
       /укажите\s+дату\s+рождения/i -> 'birthdate'
       /отсутствует\s+в\s+реестре|состоит\s+в\s+реестре/i -> 'result'
       /поиск\s+иностранца/i -> 'intro'
       /загружается|подождите|проверка\s+данных/i -> 'unknown' (loading)
     - Если 'unknown' -> ожидание 2 сек, повтор

  2. ШАГ INTRO: НАЧАЛО
     - Клик кнопки "Начать" (через MAIN world executeScript)
     - waitForAnyFormStep() --- ожидание появления любого шага формы (до 8 сек)

  3. ШАГ BIRTHDATE: ЗАПОЛНЕНИЕ ДАТЫ РОЖДЕНИЯ
     - Найти input (name="c_birth_date" или label "Дата")
     - Заполнить через execCommand('insertText') в MAIN world
     - Пауза 300-500 мс (имитация человека)
     - Клик кнопки "Продолжить" (MAIN world)
     - waitForStepChange('birthdate') --- ожидание смены шага (до 8 сек)

  4. ШАГ DOCUMENT: ЗАПОЛНЕНИЕ ДОКУМЕНТА
     - Найти input для "Серия" -> заполнить через execCommand (MAIN world)
     - Найти input для "Номер" -> заполнить
     - Найти input для "Дата выдачи" -> заполнить
     - Пауза 300-500 мс
     - Клик кнопки "Продолжить" (MAIN world)
     - waitForStepChange('document') --- ожидание смены шага (до 8 сек)

  5. ШАГ RESULT: ЧТЕНИЕ РЕЗУЛЬТАТА
     - Прочитать текст результата в DOM (content script)
     - Определить: "Отсутствует" = not_found, иначе = found
     - Считать дополнительные данные (документ, дата, время МСК)
     - Отправить результат в background worker
     - Пауза 3-5 сек (настраиваемая)
     - Клик кнопки "Проверить ещё" (MAIN world)
     - Ожидание возврата к intro/birthdate (до 8 сек)

  6. ИНКРЕМЕНТ И ПЕРЕХОД
     - Background worker обновляет прогресс в popup
     - Переход к следующему сотруднику -> начало while loop
```

**Обработка ошибок:**

| Ошибка | Действие |
|---|---|
| Элемент не найден (таймаут) | Пометить сотрудника как "ошибка", перейти к следующему |
| Кнопка "Продолжить" неактивна | Повторить заполнение полей, если не помогло --- ошибка |
| Госуслуги показали ошибку | Скриншот текста ошибки в лог, повторная попытка (до 3 раз) |

| Потеря связи с сервером лицензий | Продолжить работу (кэш лицензии), синхронизировать счётчик позже |
| Пользователь закрыл вкладку | Остановить проверку, сохранить промежуточные результаты |

### 6.5. Управление проверкой

- **Пауза** --- остановить после текущего сотрудника, не начинать следующего
- **Продолжить** --- возобновить с того места где остановились
- **Остановить** --- прервать полностью, результаты сохранены
- **Повторить ошибочные** --- запустить проверку заново только для сотрудников со статусом "ошибка"

### 7.3. Content Script --- gosuslugi.ts (подробная спецификация)

Content script работает в **ISOLATED world** и выполняет **только чтение** DOM. Все операции записи (заполнение полей, клики) выполняются из background worker через `chrome.scripting.executeScript({ world: 'MAIN' })`.

**Заполнение полей Angular --- execCommand('insertText') в MAIN world:**

Angular/EPGU использует `DefaultValueAccessor`, который слушает `(input)` event. Content script из ISOLATED world создаёт untrusted события (`isTrusted=false`), которые Angular игнорирует. `document.execCommand('insertText')` создаёт trusted input events.

```typescript
// Выполняется через chrome.scripting.executeScript({ world: 'MAIN' })
function setVal(selector: string, value: string): void {
  const el = document.querySelector<HTMLInputElement>(selector);
  if (!el) throw new Error(`Input not found: ${selector}`);

  el.focus();
  el.select();

  // Strategy 1: execCommand creates trusted input events
  const inserted = document.execCommand('insertText', false, value);
  if (inserted && el.value === value) { el.blur(); return; }

  // Strategy 2: native setter + events (fallback)
  const setter = Object.getOwnPropertyDescriptor(
    window.HTMLInputElement.prototype, 'value'
  )?.set;
  if (setter) setter.call(el, value);
  else el.value = value;

  el.dispatchEvent(new Event('input', { bubbles: true }));
  el.dispatchEvent(new Event('change', { bubbles: true }));
  el.blur();
}
```

**Для полей даты (lib-date-picker):**

Госуслуги используют EPGU-компонент `lib-date-picker`. Формат ввода: ДД.ММ.ГГГГ. Заполнение работает через прямой ввод в текстовое поле `<input>` внутри компонента тем же методом `execCommand('insertText')`. Дата рождения: `name="c_birth_date"`, label "Дата".

function waitForElement(selector: string, textContent?: string, timeout = 30000): Promise<Element> {
  return new Promise((resolve, reject) => {
    const existing = findElement(selector, textContent);
    if (existing) return resolve(existing);

    const observer = new MutationObserver(() => {
      const el = findElement(selector, textContent);
      if (el) { observer.disconnect(); resolve(el); }
    });
    observer.observe(document.body, { childList: true, subtree: true, characterData: true });
    setTimeout(() => { observer.disconnect(); reject(new Error(`Timeout: "${selector}" not found`)); }, timeout);
  });
}

### 8.4. Привязка ключа

Ключ **не привязан к устройству** (device_id не используется). Один ключ может использоваться на нескольких устройствах одновременно. Счётчик `used_this_month` общий для всех устройств --- это ограничивает злоупотребление при передаче ключа.

### 8.5. Сброс счётчика

Cron-задача: каждый 1-го числа месяца в 00:00 МСК обнуляет used_this_month для всех активных лицензий.

### 8.6. Безопасность

- HMAC-SHA256 подпись ответов (секретный ключ вшит в расширение, обфусцирован)
- Rate limiting: не более 60 запросов/мин с одного IP
- Множественные устройства на один ключ (без device_id привязки)
- HTTPS обязателен
- Логирование всех запросов (без ПДн)

## 11. Юридические аспекты

### 11.1. 152-ФЗ --- Персональные данные

Расширение RKL Check **не является оператором персональных данных** в смысле 152-ФЗ:

- Не собирает ПДн сотрудников клиента
- Не хранит ПДн на своих серверах
- Не передаёт ПДн третьим лицам
- Данные обрабатываются исключительно в браузере пользователя
- Передача данных происходит только на Госуслуги --- государственную информационную систему, через авторизованный запрос от имени организации-работодателя

Расширение является **инструментом автоматизации** --- эквивалент макроса, который делает то же самое что кадровик делает руками, но быстрее.

Организация-работодатель (клиент) самостоятельно несёт ответственность за:
- Наличие правовых оснований для проверки сотрудников по РКЛ
- Получение согласия на обработку ПДн (или наличие иного законного основания)
- Обеспечение безопасности данных на своих устройствах

### 11.2. Оферта и условия использования

Необходимо подготовить:

1. **Публичная оферта** --- договор на оказание услуг (предоставление доступа к ПО)
2. **Политика конфиденциальности** --- описание обработки данных (email организации, платёжные данные)
3. **Условия использования расширения** --- отказ от ответственности за действия пользователя, правила использования

### 11.3. Юридическое лицо

Для приёма платежей от юрлиц и оформления оферты потребуется:
- ИП или ООО
- Расчётный счёт
- Договор с платёжной системой

## 13. План разработки

### Фаза 1 --- MVP (ЗАВЕРШЕНА)

Цель: работающее расширение без лицензирования

Реализовано:

- Content script (ISOLATED world): определение шагов формы, чтение результата, обнаружение rate limit
- MAIN world скрипты: заполнение полей через `execCommand('insertText')`, клики по кнопкам
- Адаптивный цикл проверки: обрабатывает шаги в любом порядке (while loop), двухфазный waitForStepChange
- Popup: 5 экранов (лицензия, загрузка Excel, предпросмотр, прогресс, результаты)
- Background worker: state machine, очередь, обмен сообщениями
- Экспорт результатов в Excel (SheetJS)
- UI: профессиональный дизайн Navy/Slate (#0F172A, #0369A1, #F8FAFC)
- Полноэкранная страница результатов (result.html): таблица 9 колонок, фильтры-табы со счётчиками, поиск, сортировка, экспорт

- Обработка rate limit Госуслуг: остановка проверки + chrome.notifications
- Предупреждение в настройках при задержке < 10 сек
- Обработка loading-страниц между шагами формы
- Протестировано на реальных данных через Госуслуги

## 14. Технические ограничения и допущения

1. Пользователь должен быть авторизован на Госуслугах как сотрудник организации до начала проверки
2. Вкладка с Госуслугами должна быть активной и видимой во время проверки
3. Скорость проверки ограничена Госуслугами --- каждая проверка занимает 5-30 секунд
4. Расширение не может работать в фоне без открытой вкладки Госуслуг
5. URL формы проверки РКЛ: `https://www.gosuslugi.ru/655781/1/form` --- SPA на **Angular + EPGU**, URL не меняется между шагами
6. **Порядок шагов: intro → birthdate → document → result** (НЕ document → birthdate)
7. Поля формы: Серия (необязательное), Номер, Дата выдачи, Дата рождения (`name="c_birth_date"`, label "Дата"). ФИО не требуется и не передаётся
8. Задержка между проверками: настраиваемая 2-15 сек, по умолчанию 10 сек (при <10 сек — предупреждение о risk rate limit от Госуслуг)
9. Формат дат на Госуслугах: ДД.ММ.ГГГГ
10. **Angular-поля заполняются через `document.execCommand('insertText')` в MAIN world** --- создаёт trusted input events. Fallback: nativeInputValueSetter + dispatchEvent
11. Между шагами формы бывают **loading-страницы** ("Загружается", "Подождите", "Проверка данных") --- обрабатываются как `unknown`, двухфазный waitForStepChange
12. Формулировка при нахождении в РКЛ: **"Состоит в реестре контролируемых лиц"** (подтверждено). При отсутствии: "Отсутствует в реестре контролируемых лиц"
13. Поведение при неполной дате рождения (чекбокс) не тестировалось --- требует уточнения
14. Максимальная рекомендуемая партия: 200-300 сотрудников за одну сессию (для минимизации риска rate limit)
15. **Rate limit:** при задержке < 10 сек Госуслуги могут показать страницу "Превышено количество запросов" → расширение останавливает проверку, уведомляет пользователя
16. **Пре-валидация:** строки без обязательных полей (номер, дата выдачи, дата рождения) помечаются как `error` при загрузке Excel и не отправляются на Госуслуги
17. **Двухфазный waitForStepChange:** Phase 1 (stepTimeout=10 сек) — быстрый таймаут для валидации формы; Phase 2 (resultTimeout=120 сек) — только если обнаружена loading-страница (запрос к МВД)
